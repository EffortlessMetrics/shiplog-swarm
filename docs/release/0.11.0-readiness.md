# shiplog 0.11.0 - Release Readiness

**Release target:** `v0.11.0`
**Theme:** low-friction review readiness
**Status:** release preparation

## Included state

- Objective-scoped setup readiness keeps missing share configuration from
  blocking local intake.
- GitHub can reuse supported environment credentials or an authenticated `gh`
  session without persisting secrets.
- `shiplog`, `shiplog next`, `shiplog update`, `shiplog add`, and default
  `shiplog open` provide the normal read-only, refresh, capture, and packet
  workflow.
- Direct manager/public share commands explain and verify their profile before
  rendering, while remaining fail-closed.
- Versionless installers, Homebrew, and Scoop provide prebuilt installation
  paths.
- Routine CI remains Linux-first; release builds retain Linux, Windows, and
  macOS asset coverage.

## Required preflight

```text
rtk cargo fmt --all -- --check
rtk cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
rtk cargo test --workspace --all-features --locked
rtk cargo xtask check-no-panic-family --mode blocking-allowlist
rtk cargo xtask check-policy-schemas
rtk cargo xtask check-file-policy --mode blocking-allowlist
rtk cargo xtask check-generated --mode blocking-allowlist
rtk cargo publish -p shiplog --dry-run --locked
rtk bash scripts/package-proof.sh
rtk bash scripts/publish-dry-run.sh
rtk git diff --check
```

## Release execution

Tagging, publishing, and promoting the GitHub release require explicit owner
approval after the preflight and routed CI result pass:

```text
rtk git tag -a v0.11.0 -m "shiplog v0.11.0"
rtk git push origin v0.11.0
rtk cargo publish -p shiplog --locked
rtk gh release edit v0.11.0 --draft=false --latest
```

Post-release proof must verify the crate, GitHub assets/checksums, installers,
Homebrew, Scoop, and a packaged first-packet smoke without Rust installed.
