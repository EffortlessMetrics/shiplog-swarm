## Purpose

This receipt records the crate surface that shipped during `0.6.0` and the
forward support stance for the 0.7 crate-surface contraction lane.

`0.6.0` made the first-run review-pack path trustworthy. It also published
implementation crates that are not intended to be long-term crates.io support
surfaces. This document names that state without rewriting it.

## Compatibility

Do not yank the 0.6 implementation crates as routine cleanup. Yanking is
reserved for security issues or severe correctness problems.

The safer compatibility posture is:

- existing 0.6 crates remain available for historical dependency resolution;
- new 0.7+ release tooling publishes only the explicit allowlist;
- unsupported implementation crates are not documented as forward public import
  surfaces;
- a real external Rust consumer can reopen a crate through an ADR-backed public
  support decision.

## User Guidance

For users installing shiplog:

```bash
cargo install shiplog
```

For machine-readable contracts, prefer the JSON schemas under
`contracts/schemas/` unless an ADR explicitly keeps `shiplog-schema` public.

For Rust consumers that imported 0.6 implementation crates directly, pin the
0.6 dependency while opening an issue that explains the needed public contract.
Do not infer support from the fact that the 0.6 crate was published.

## Proof And Controls

The 0.7 lane controls the forward surface through:

- [`policy/publish-allowlist.toml`](../../policy/publish-allowlist.toml);
- [`scripts/package-boundary-audit.sh`](../../scripts/package-boundary-audit.sh);
- [`scripts/publish-dry-run.sh`](../../scripts/publish-dry-run.sh).

Those controls are forward-looking. They do not change the historical fact that
0.6 crates exist on crates.io.
