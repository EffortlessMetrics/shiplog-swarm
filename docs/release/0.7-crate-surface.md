## Purpose

This audit classifies every current workspace package for the 0.7.0
crate-surface contraction lane.

It is a release control surface, not an implementation plan. It records which
packages are supported public contracts, which are transitional decisions, and
which are implementation or development support that should not be published by
0.7 release tooling.

## Current State

The workspace currently contains 22 packages:

- `shiplog`;
- 20 packages under `crates/`;
- `xtask`.

The historical 0.6.0 publish script lists 22 publishable packages: `shiplog`
plus every workspace package that existed then except `shiplog-testkit` and
`xtask`.

Current manifest posture:

- `shiplog-testkit` has `publish = false`;
- `xtask` has `publish = false`;
- `shiplog-schema` and every current `internal-module` package have
  `publish = false` in their Cargo manifests;
- `shiplog-render-md` and `shiplog-render-json` have been removed as workspace
  packages and inlined as owner modules for the first contraction slice;
- `policy/publish-allowlist.toml` limits the 0.7 release publish order to
  `shiplog` by default;
- `scripts/package-boundary-audit.sh`, `scripts/package-proof.sh`, and
  `scripts/publish-dry-run.sh` consume that allowlist instead of hard-coding
  workspace members;
- package-boundary audit fails if a policy-disabled package is manifest
  publishable.

That is the mismatch this lane must correct. Useful internal boundaries should
not become crates.io support promises by default.

## Target Public Surface

Default supported 0.7 public surface:

```text
shiplog
```

Pending explicit decision:

```text
shiplog-schema
```

`shiplog-schema` remains public only if an ADR records a typed Rust contract
reason or a known external Rust consumer. Otherwise, JSON schemas under
`contracts/schemas/` remain the public machine contract.

## Immediate Mismatches

The audit identifies current-state gaps for follow-up PRs:

1. The historical 0.6 publish script lists implementation crates.
2. The renderer crates have been collapsed, but most remaining internal packages
   are still separate workspace packages.
3. `shiplog` still has normal path dependencies on internal workspace packages
   until those packages are collapsed into owner modules or otherwise removed
   from the public install dependency graph.
4. `shiplog-schema` still needs a keep-or-inline ADR before release prep.

This document names the gaps so implementation and release-tooling PRs have a
stable target.

## Required Proof By Change Type

### Docs And Planning PRs

```bash
cargo xtask check-policy-schemas
cargo xtask check-file-policy --mode blocking-allowlist
git diff --check
```

### Render Collapse PRs

```bash
cargo test -p shiplog --test render_md
cargo test -p shiplog-engine --test render_json
cargo test -p shiplog --test cli_integration -- report
cargo test -p shiplog --test intake_cold_start
git diff --check
```

Rendered Markdown and JSON output shape must remain unchanged unless a separate
spec-backed behavior PR changes it.

### Ingest Adapter Collapse PRs

```bash
cargo test -p shiplog --test intake_cold_start
cargo test -p shiplog --test front_door_first_pack_smoke
cargo test -p shiplog-ingest-github recorded_http_fixtures_prove_full_fresh_then_cached_ingest
git diff --check
```

Source identity, skipped freshness, stale freshness, and recorded GitHub
fixture behavior must remain unchanged. After the GitHub adapter is collapsed,
the same fixture proof should move to the new package/module location rather
than disappear.

### Product And Trust Support Collapse PRs

Use the smallest relevant subset first, then broaden when the touched surface
crosses report, cache, redaction, bundle, or CLI behavior:

```bash
cargo test -p shiplog --test intake_cold_start
cargo test -p shiplog --test front_door_first_pack_smoke
cargo test -p shiplog --test cli_integration -- intake
cargo test -p shiplog --test cli_integration -- report
cargo test -p shiplog-cache
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo xtask check-policy-schemas
cargo xtask check-file-policy --mode blocking-allowlist
git diff --check
```

When a package named in this command set has already been collapsed, use the
equivalent test in its new module/package location and record that mapping in
the PR body.

### Release Tooling PRs

Release tooling must prove:

- the publish allowlist is explicit;
- only `public-supported` crates are publishable by default;
- `public-transitional` crates require a named exception;
- internal, dev-only, and historical crates cannot enter the 0.7 publish
  sequence accidentally;
- policy-disabled workspace packages have `publish = false` in their Cargo
  manifests;
- `cargo install shiplog` remains the user path.

Current control surface:

```bash
scripts/package-boundary-audit.sh
scripts/publish-dry-run.sh
```

## Compatibility Notes

0.6.0 implementation crates are historical/transitional. Do not yank them as
routine cleanup. The exact 0.6 crate list and forward support posture are
recorded in
[`0.6-crate-surface-historical`](./0.6-crate-surface-historical.md).

0.7.0 should stop publishing unsupported implementation surfaces forward. If a
real external Rust consumer depends on a non-CLI crate, record that fact and
promote the crate through an ADR instead of treating the 0.6 publication as
implicit proof of support.

## Open Decisions

- Decide whether `shiplog-schema` remains public or becomes internal.
- Decide whether collapsed crates are removed from the workspace immediately or
  remain as unpublished transitional packages while dependent modules are moved.
- Continue collapsing or marking non-public implementation packages after the
  release tooling guardrail.

## Non-Goals

This audit does not:

- change first-run behavior;
- change report schemas;
- start Evidence Repair Loop work;
- create a plugin or adapter API;
- yank 0.6.0 crates.
