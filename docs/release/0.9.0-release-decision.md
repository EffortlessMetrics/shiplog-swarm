**Date:** 2026-05-17

`main`

## Decision

Keep the `v0.9.0` release hold active.

This is the post-harvest release decision after GitHub activity harvest
completed through status/report/merge, resume proof, completion audit, and
redaction alias correctness.

Do not tag, publish to crates.io, create a GitHub release, dispatch the release
workflow, or run release-install smoke for `v0.9.0` from this decision.

The product loop is feature-complete enough to consider a release-resume PR:
setup preflight, status cockpit, intake, repair, rerun, repair diff, runs diff,
share explanation, and budgeted GitHub activity harvest now form one
receipt-backed operating loop. That does not authorize release execution. The
missing release inputs remain explicit owner approval and current final
preflight from merged `main`.

## Current Release Posture

- `v0.8.0` remains the latest shipped release.
- crates.io latest remains `shiplog = "0.8.0"`.
- `v0.9.0` remains a prepared but unreleased candidate, now including
  review-ready packet quality, Guided Setup / Doctor, Review-loop Status,
  GitHub activity harvest, HMAC redaction correctness, and release-facing docs
  on `main`.
- `scripts/check-release-hold.sh` blocks held `v0.9.0` release execution.
- Manual `release.yml` dispatch requires a semver tag and
  `owner_approved_release_execution`, and the active hold still blocks
  `v0.9.0`.

## Resume Conditions

Open a future release-resume PR only when all of these are true:

- the owner explicitly approves release execution;
- `docs/release/0.9.0-release-hold.md` is updated to lift or replace the hold;
- review-ready, Guided Setup / Doctor, Review-loop Status, and GitHub activity
  harvest caveats are reviewed;
- HMAC redaction alias correctness and cached-alias compatibility are reviewed;
- current `main` CI is green;
- package-version, package-boundary, publish dry-run, product proof, docs/share
  and status/runs CLI tests, clippy, policy gates, no-panic, and install smoke
  are rerun from current `main`;
- no local or remote `v0.9.0` tag exists before the approved release action.

## What To Do Next

Do not add features just to justify 0.9.

Allowed next work:

- fix concrete dogfood bugs found in the current loop;
- refresh release docs when a proof receipt changes;
- prepare a future release-resume PR only after explicit approval.

Do not extend Guided Setup / Doctor, Review-loop Status, or GitHub activity
harvest just to make 0.9 larger. The candidate is represented by review-ready
packet quality, Guided Setup / Doctor, Review-loop Status, GitHub activity
harvest, redaction correctness, and release-facing docs. Further work should be
bug-driven or deliberately opened as a new non-release lane.
