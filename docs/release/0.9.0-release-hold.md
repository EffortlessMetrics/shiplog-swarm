# shiplog 0.9.0 - Release Hold

**Status:** paused
**Hold date:** 2026-05-16
**Last shipped release:** `v0.8.0`
**Candidate version metadata:** `0.9.0`
**Current decision:** keep hold ([release decision](0.9.0-release-decision.md))

## Decision

Do not tag, publish, or otherwise release `v0.9.0` now.

`v0.8.0` shipped the Evidence Repair Loop to GitHub and crates.io. The next
stretch is post-0.8 hardening and UX soak, not another immediate release train.
The review-ready packet, Guided Setup / Doctor, Review-loop Status, GitHub
activity harvest, and redaction correctness work that landed after 0.8 stays on
`main` as an unreleased 0.9 candidate.

## Current State

- `v0.8.0` is the latest shipped release.
- PR #319 prepared 0.9.0 version metadata and release docs, but did not tag,
  publish to crates.io, or publish a GitHub release.
- `packet_quality`, Packet Readiness, claim candidates, share explain, runs
  diff, the product proof, and the guide remain available on `main`.
- Guided Setup / Doctor work also remains available on `main`: `init
  --guided`, `doctor --setup`, `sources status`, setup-blocked repair
  handoffs, share-profile setup readiness, product proof, and guide.
- Review-loop Status work also remains available on `main`: `status --latest`,
  `status --latest --json`, status schema/examples, consistency proof,
  safe-next-action proof, dogfood transcript, recurring guide, and README
  alignment.
- GitHub activity harvest work also remains available on `main`: `plan`,
  `scout`, `run --profile authored --resume`, `run --profile full --resume`,
  `status`, `report`, `merge`, plan/progress/API-ledger/report schemas,
  owner-filter receipts, API-cost receipts, resume proof, guide, and
  completion audit.
- Deterministic redaction aliases now use the documented HMAC-SHA256 primitive
  for new uncached aliases while preserving existing cached aliases.
- Workspace package versions remain `0.9.0` as candidate metadata.
- The tag-triggered and manually dispatched release workflow runs
  `scripts/check-release-hold.sh` before release proof; while this receipt
  exists, an accidental `v0.9.0` release fails before package proof, binary
  builds, or GitHub release creation.
- Manual `release.yml` dispatch requires an explicit semver `release_tag` and
  `owner_approved_release_execution`; branch refs such as `main` are rejected
  before publish dry-run proof can start.

## Blocked Operations

While this hold is active, do not run:

- `git tag v0.9.0`
- `git push origin v0.9.0`
- `cargo publish -p shiplog --locked`
- `gh release edit v0.9.0 --draft=false --latest`
- manual `release.yml` dispatch for `v0.9.0`
- release-install smoke against `v0.9.0` assets

These operations require a fresh release-resume decision and current proof.

## Soak Focus

- dogfood the 0.8 repair loop on real review flows;
- run review-ready packet flows locally and file small bugfix PRs;
- improve wording, help, docs, and tests around packet quality;
- tighten no-write behavior for share explain and runs diff;
- keep advisory proof surfaces separate from release blocking;
- avoid new public API churn unless a real bug requires it.

## Resume Criteria

Resume 0.9 release execution only for a concrete reason:

- a security fix;
- broken 0.8 behavior;
- a major UX bug in the shipped repair loop;
- enough validated post-0.8 value to justify another release.

A future release-resume PR should update this hold receipt, record owner
approval, rerun release proof from current `main`, and replace paused handoff
wording with copy-ready release instructions.
